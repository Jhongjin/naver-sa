# Naver SA Autopilot 운영자 기능 설명서

이 문서는 같은 이름의 PPTX 슬라이드 원본 텍스트입니다.
실제 운영자가 플랫폼을 검수할 때 기능 범위, 사용 순서, 확인 포인트를 한 장씩 따라볼 수 있도록 작성했습니다.

절대 정책: 라이브 집행 금지, 삭제 금지, 승인된 테스트 생성/수정만 허용, 민감값 비공개.

## 1. 운영자 기능 설명서

대주제: Naver SA Autopilot

파워링크/사이트검색광고와 쇼핑검색 MVP 자동화 흐름을 한 장씩 따라보는 검수용 슬라이드입니다.

### 구현된 기능
- 대상: 실제 광고 운영자, 관리자, 사용자 검수 담당자.
- 원칙: 라이브 집행 금지, 삭제 금지, 승인된 테스트 생성/수정만 허용.
- 목표: 플랫폼을 써 보며 추가로 필요한 기능과 보완점을 정확히 찾을 수 있게 돕습니다.

### 운영자가 확인할 점
- 플랫폼을 함께 열어두고 슬라이드 순서대로 한 기능씩 확인합니다.
- 광고비가 나가거나 실서비스가 바뀌는 동작은 MVP에서 막혀 있어야 정상입니다.

## 2. 이 슬라이드를 보는 방법

대주제: 문서 안내

기능을 개발자 설명이 아니라 운영자가 실제로 판단할 순서로 정리했습니다.

### 구현된 기능
- 각 슬라이드는 무엇을 하는 기능인지, 언제 쓰는지, 확인할 점을 분리해 설명합니다.
- 화면에서 보이는 용어는 최대한 그대로 사용하되, 어려운 표현은 쉬운 말로 풀었습니다.
- 검수 중 헷갈리는 이름, 빠진 버튼, 더 필요한 필터는 별도로 기록하면 다음 개선 큐가 됩니다.

### 운영자가 확인할 점
- 먼저 전체 흐름을 훑고, 두 번째 회독에서 실제 계정과 샘플 브랜드로 기능을 눌러보는 방식이 좋습니다.
- 고객에게 설명하기 어려운 문구는 기능 부족이 아니라 표현 개선 과제로 남겨도 됩니다.

## 3. 플랫폼이 해결하려는 일

대주제: 제품 개요

광고 세팅과 운영 검토를 한 명의 운영자가 한 흐름에서 처리하도록 돕는 보조 플랫폼입니다.

### 구현된 기능
- 초기 세팅: 브랜드, 사이트, 예산, 목표를 입력하면 키워드와 광고 구조 초안을 만듭니다.
- 검수: 자동으로 만든 항목을 바로 보내지 않고 승인, 보류, 메모로 나눠 확인합니다.
- 연결: 네이버 계정에서 비즈채널, 캠페인, 쇼핑 상품그룹 후보를 읽어와 초안에 연결합니다.
- 보관: 어떤 항목을 승인했고 어떤 초안이 만들어졌는지 이력과 리포트로 남깁니다.

### 운영자가 확인할 점
- 이 플랫폼은 광고를 대신 집행하는 도구가 아니라, 운영자가 승인하기 전까지 멈춰 있는 세팅 보조 도구입니다.

## 4. 현재 MVP 범위

대주제: 제품 개요

우선 완성도를 높이는 대상은 파워링크/사이트검색광고와 쇼핑검색 세팅 보조입니다.

### 구현된 기능
- 파워링크/사이트검색광고: 사이트 비즈채널을 기준으로 캠페인, 광고그룹, 키워드, 소재 초안을 만듭니다.
- 쇼핑검색: 쇼핑몰 비즈채널과 상품그룹 연결을 확인한 뒤 쇼핑 캠페인, 광고그룹, 검색어 초안을 만듭니다.
- 성과 조회: 저장된 조회 계획을 바탕으로 읽기 전용 성과 확인과 운영 추천을 만듭니다.
- 관리자 기능: 사용자, 운영 상태, 계정 스캔, 공유 링크, 감사 이벤트, 예약 실행 상태를 관리합니다.

### 운영자가 확인할 점
- 브랜드검색, 플레이스 광고 전체 자동화, 라이브 입찰 자동화는 현재 범위 밖입니다.
- 쇼핑 상품그룹 생성이나 쇼핑 피드 수정도 현재 플랫폼에서 직접 하지 않습니다.

## 5. 운영자가 하루에 따라가는 기본 순서

대주제: 전체 흐름

처음부터 끝까지 한 줄로 보면 입력, 검수, 연결, 저장, 보고, 관리입니다.

### 구현된 기능
- 1단계: 로그인 후 작업공간에서 상품 유형과 브랜드 정보를 입력합니다.
- 2단계: 키워드, 광고그룹, 예산/입찰, 제외 후보를 확인합니다.
- 3단계: 승인 큐에서 보낼 항목만 승인하고, 애매한 항목은 보류 메모를 남깁니다.
- 4단계: 계정 스캔으로 비즈채널, 캠페인, 쇼핑 상품그룹을 읽어와 연결합니다.
- 5단계: 전송 직전 초안을 검증하고, 이력 저장 후 내부/외부 보고 자료로 공유합니다.

### 운영자가 확인할 점
- 검수 중 가장 중요한 질문은 이 항목이 실제 광고 계정에 들어가도 되는가입니다.
- 대답이 애매하면 승인하지 말고 보류 메모로 남기는 것이 안전합니다.

## 6. 자동화 단계와 현재 허용 범위

대주제: 운영 원칙

MVP는 읽기, 초안 생성, 검증, 제한된 테스트 생성까지만 허용하는 구조입니다.

### 구현된 기능
- 읽기 전용: 계정, 캠페인, 비즈채널, 상품그룹, 성과를 조회합니다.
- 초안 전용: 키워드, 광고그룹, 소재, 예산, 운영 추천을 작성하지만 네이버에 보내지 않습니다.
- 전송 직전 초안: 승인된 항목만 실제 요청 직전 형태로 변환해 검토합니다.
- 보호된 테스트 실행: 관리자 권한, 저장된 준비 완료 초안, 확인 문구가 모두 맞을 때만 허용합니다.

### 운영자가 확인할 점
- 라이브 광고 활성화와 삭제는 현재 버전에서 허용되지 않습니다.
- 실수로 바로 집행되는 흐름이 보이면 기능 결함으로 기록해야 합니다.

## 7. 절대 바뀌면 안 되는 안전선

대주제: 운영 원칙

운영자가 안심하고 검수할 수 있도록 위험한 동작은 기본적으로 막아둡니다.

### 구현된 기능
- 라이브 집행 금지: 광고비가 나가는 실제 활성화는 MVP에서 하지 않습니다.
- 삭제 금지: 지우기 대신 중지/보관 방향만 허용합니다.
- 승인 우선: 생성, 수정, 입찰, 예산, 제외 키워드 추가는 사람의 승인 기록이 필요합니다.
- 민감값 비공개: 키, 토큰, 내부 식별자, 원본 오류 문구는 화면과 보고서에 노출하지 않습니다.

### 운영자가 확인할 점
- 검수자는 실수해도 광고 계정이 망가지지 않는가를 계속 확인해야 합니다.
- 승인 전송보다 기록과 차단 상태가 더 중요합니다.

## 8. 로그인, 가입, 세션 확인

대주제: 접속과 권한

현재 운영자 접근은 Supabase Auth 기반입니다. 예전 운영자 코드 방식은 폐기 방향입니다.

### 구현된 기능
- 로그인 화면에서 이메일 기반 계정으로 접속합니다.
- 가입 후 메일 확인이 필요한 상태를 화면에서 확인하고 재시도할 수 있습니다.
- 내 계정 상태는 세션 확인 화면과 마이페이지에서 볼 수 있습니다.
- 예전 운영자 세션 API는 더 이상 실제 사용 경로가 아니며 폐기된 경로로 남아 있습니다.

### 운영자가 확인할 점
- 로그인 실패 메시지가 너무 기술적이거나 불안한 문구인지 확인합니다.
- 메일 확인, 재로그인, 권한 부족 상태에서 안내가 충분한지 살펴봅니다.

## 9. 사용자 권한 구분

대주제: 접속과 권한

일반 멤버와 관리자는 볼 수 있는 범위와 실행 가능한 운영 작업이 다릅니다.

### 구현된 기능
- 멤버: 본인 작업공간에서 계획을 만들고, 승인 상태를 저장하고, 자신의 이력을 확인합니다.
- 관리자: 사용자 관리, 네이버 계정 스캔, 성과 조회, 운영 알림, 전체 활동 확인을 수행합니다.
- 관리자 전용 기능은 외부 계정 정보를 읽을 수 있으므로 일반 멤버에게 열지 않습니다.
- 각 화면은 권한이 부족할 때 데이터를 보여주지 않고 안내 메시지로 막아야 합니다.

### 운영자가 확인할 점
- 일반 멤버 계정으로 관리자 화면이나 네이버 계정 스캔이 열리지 않는지 확인합니다.
- 완전한 다계정 역할 분리와 계정별 네이버 인증 전환은 아직 확장 과제입니다.

## 10. 마이페이지와 작업공간 연결

대주제: 마이페이지

운영자는 본인의 계정 상태와 연결된 작업공간을 확인한 뒤 세팅 화면으로 들어갑니다.

### 구현된 기능
- 로그인 계정, 메일 확인 상태, 워크스페이스 연결 상태를 확인합니다.
- 세션이 불안정할 때 다시 시도하거나 로그인 화면으로 돌아갈 수 있습니다.
- 워크스페이스 소유 여부는 서버에서 판단하며 내부 사용자 식별자는 화면에 노출하지 않습니다.
- 작업공간은 이후 계획 저장과 이력 조회의 기준 단위가 됩니다.

### 운영자가 확인할 점
- 내가 어느 워크스페이스에서 작업 중인지 운영자에게 명확한지 확인합니다.
- 로컬 임시저장은 감사 이력이 아니며, 공식 기준은 서버 저장 이력과 감사 이벤트입니다.

## 11. 캠페인 기준 입력

대주제: 작업공간

작업공간의 첫 단계는 브랜드와 캠페인 방향을 입력해 자동 초안의 기준을 정하는 일입니다.

### 구현된 기능
- 운영 모드: 대행사형 또는 광고주형 흐름을 선택할 수 있습니다.
- 상품 유형: 파워링크 또는 쇼핑검색을 선택합니다.
- 기본 정보: 브랜드명, 사이트 주소, 업종, 월 테스트 예산, 최대 입찰 기준을 입력합니다.
- 시드 키워드: 운영자가 생각하는 출발 키워드를 넣으면 확장 후보가 생성됩니다.
- 로컬 임시저장: 입력값, 승인 상태, 연결 ID를 복구할 수 있습니다.

### 운영자가 확인할 점
- 입력값이 바뀌면 기존 계정 스캔 후보가 다른 브랜드의 값인지 다시 판단해야 합니다.
- 현재 화면은 인증 정보나 세션 값을 임시저장하지 않아야 합니다.

## 12. 파워링크와 쇼핑검색 선택 차이

대주제: 작업공간

상품 유형을 바꾸면 필요한 연결값과 생성되는 초안의 종류가 달라집니다.

### 구현된 기능
- 파워링크: PC/모바일 사이트 비즈채널을 기준으로 키워드, 광고그룹, 소재 초안을 만듭니다.
- 쇼핑검색: 쇼핑몰 비즈채널과 상품그룹 ID가 맞아야 쇼핑 광고그룹 초안을 안정적으로 만들 수 있습니다.
- 쇼핑검색에서는 키워드보다 상품검색어 관점의 검수가 중요합니다.
- 두 유형 모두 승인된 항목만 전송 직전 초안으로 바뀝니다.

### 운영자가 확인할 점
- 쇼핑검색 검수 때는 상품그룹이 네이버 콘솔에 이미 존재하는지 먼저 확인해야 합니다.
- 상품그룹 생성 자체는 현재 플랫폼에서 하지 않습니다.

## 13. 키워드 후보 생성과 분류

대주제: 키워드와 검색어

입력한 키워드를 바탕으로 등록 후보, 검토 후보, 제외 후보를 나눠 보여줍니다.

### 구현된 기능
- 포함 후보: 광고 등록 초안에 들어갈 수 있는 키워드 또는 검색어입니다.
- 검토 후보: 의도는 맞을 수 있지만 운영자 판단이 더 필요한 항목입니다.
- 제외 후보: 저품질 유입이나 브랜드와 맞지 않는 검색을 줄이기 위한 항목입니다.
- 키워드 CSV를 내려받아 별도 검수표나 고객 공유 자료로 사용할 수 있습니다.

### 운영자가 확인할 점
- 너무 넓은 키워드, 경쟁사/민감어, 낮은 구매 의도 키워드가 포함 후보에 섞이지 않았는지 봅니다.

## 14. 캠페인과 광고그룹 초안

대주제: 광고 구조 초안

키워드를 운영하기 쉬운 묶음으로 나누고, 각 묶음에 맞는 광고 구조를 제안합니다.

### 구현된 기능
- 캠페인 초안: 상품 유형과 운영 목적에 맞는 큰 단위를 만듭니다.
- 광고그룹 초안: 비슷한 의도나 상품군을 기준으로 묶어 관리 단위를 만듭니다.
- 소재/문안 초안: 광고그룹 성격에 맞는 문구 초안을 함께 확인합니다.
- 쇼핑검색은 쇼핑 광고그룹과 상품검색어 매핑 관점으로 표시됩니다.

### 운영자가 확인할 점
- 광고그룹 이름이 실제 운영자가 나중에 보고 이해할 수 있는지 확인합니다.
- 고객 보고용 용어와 내부 운영용 용어가 섞여 혼란스럽지 않은지도 봅니다.

## 15. 예산과 입찰 가드레일

대주제: 예산과 입찰

자동 추천이 과하게 커지지 않도록 운영자가 입력한 한도 안에서 초안을 만듭니다.

### 구현된 기능
- 월 테스트 예산을 기준으로 무리한 캠페인 예산을 막습니다.
- 최대 입찰 기준을 넘는 키워드 입찰 초안은 만들지 않도록 제한합니다.
- 라이브 활성화는 차단되어 있어 예산 추천만으로 실제 비용이 나가지 않습니다.
- 보호 실행 전에도 서버에서 승인, 연결값, 차단 조건을 다시 확인합니다.

### 운영자가 확인할 점
- 고객이 말한 테스트 예산과 화면의 예산 추천이 자연스럽게 연결되는지 확인합니다.

## 16. 업종 템플릿과 운영 규칙 보관

대주제: 운영 근거

자동 초안이 왜 그렇게 나왔는지 나중에 설명할 수 있도록 근거 정보를 함께 저장합니다.

### 구현된 기능
- 업종 템플릿: 쇼핑몰 등 업종별 기본 세팅 방향을 저장 이력에 남깁니다.
- 벤치마크 기능: 플랫폼이 참고한 운영 기준과 기능 커버리지를 보여줍니다.
- 운영 규칙: 예산, 입찰, 제외, 승인 같은 판단 기준을 이력과 메모에 남깁니다.
- 공개 리포트에는 민감한 내부 원문이 아니라 안전하게 정리된 요약만 표시됩니다.

### 운영자가 확인할 점
- 고객에게 설명할 때 왜 이 초안이 나왔는지 한 문장으로 말할 수 있는지 확인합니다.

## 17. 전송 전 검수할 작업 목록

대주제: 승인 큐

자동으로 만든 모든 변경 후보는 승인 큐에서 사람의 판단을 거칩니다.

### 구현된 기능
- 상태: 승인 대기, 승인, 보류, 차단으로 항목을 나눠 봅니다.
- 검색/필터: 많은 후보 중 필요한 항목만 빠르게 찾습니다.
- 차단 제외 승인: 이미 안전상 차단된 항목을 제외하고 한 번에 승인할 수 있습니다.
- 승인 CSV: 현재 승인 상태와 메모를 외부 검수표로 내려받을 수 있습니다.

### 운영자가 확인할 점
- 차단 항목은 실행하면 안 되는 항목입니다. 승인 버튼보다 차단 사유를 먼저 확인해야 합니다.
- 보류 메모는 나중에 고객 또는 다른 운영자가 왜 제외했는지 이해하는 기록입니다.

## 18. 승인, 보류, 메모의 의미

대주제: 승인 큐

승인 큐는 단순 체크리스트가 아니라 운영 판단을 남기는 기록 장치입니다.

### 구현된 기능
- 승인: 전송 직전 초안으로 바꿔도 되는 항목입니다.
- 보류: 현재는 넣지 않지만 이유를 남겨 재검토할 항목입니다.
- 대기: 아직 판단하지 않은 항목이며 전송 초안에 들어가지 않습니다.
- 메모: 보류 사유, 승인 근거, 고객 확인 필요사항을 자유롭게 기록합니다.

### 운영자가 확인할 점
- 승인 상태가 저장 이력 상세에서도 그대로 확인되는지 봅니다.
- 다른 운영자가 보아도 판단 이유를 이해할 수 있을 정도로 메모가 남는지 확인합니다.

## 19. 승인에서 이력 저장까지 실행 레일

대주제: 작업 순서 안내

화면 상단의 진행 레일은 다음에 무엇을 해야 하는지 알려주는 운영 내비게이션입니다.

### 구현된 기능
- 승인 확정: 승인할 항목이 있는지 먼저 확인합니다.
- 계정 스캔: 네이버 계정에서 필요한 연결 후보를 읽어옵니다.
- 초안 검증: 승인과 연결 정보를 서버에서 다시 계산합니다.
- 이력 저장: 승인 상태, 연결값, 전송 직전 초안, 차단/경고를 저장합니다.

### 운영자가 확인할 점
- 현재 막힌 이유가 화면에서 즉시 이해되는지 확인합니다.
- 다음 버튼이 무엇을 바꾸는지 불안하면 문구 개선 대상으로 기록합니다.

## 20. 네이버 계정 스캔

대주제: 계정 스캔

관리자가 네이버 계정의 캠페인, 비즈채널, 상품그룹 후보를 읽기 전용으로 불러옵니다.

### 구현된 기능
- 파워링크에서는 사이트 비즈채널 후보와 캠페인 후보를 확인합니다.
- 쇼핑검색에서는 쇼핑몰 비즈채널과 상품그룹 후보를 함께 확인합니다.
- 스캔 결과는 감사용으로 스냅샷 이력에 저장될 수 있습니다.
- 이전 스캔과 비교해 채널, 캠페인, 상품그룹 수 변화 요약을 볼 수 있습니다.

### 운영자가 확인할 점
- 이 기능은 외부 계정 정보를 읽으므로 관리자 권한에서만 실행되는 것이 정상입니다.
- 스캔은 조회만 하며 네이버 계정에 새 항목을 만들거나 지우지 않습니다.
- 현재는 운영 검수에 필요한 후보 조회이며, 전체 계정 인벤토리의 완전한 장기 보관으로 표현하면 안 됩니다.

## 21. 파워링크 비즈채널 연결

대주제: 연결값

전송 직전 초안을 만들려면 실제 네이버 계정의 사이트 비즈채널과 연결되어야 합니다.

### 구현된 기능
- PC 채널 ID와 모바일 채널 ID를 적용합니다.
- 캠페인 ID가 이미 있으면 연결하고, 없으면 생성 초안을 검토합니다.
- 스캔된 후보에서 적용하면 오타나 다른 계정 값 사용 위험을 줄일 수 있습니다.
- 직접 입력도 가능하지만 운영자가 값의 출처를 더 꼼꼼히 확인해야 합니다.

### 운영자가 확인할 점
- PC/모바일 채널이 같은 브랜드와 사이트에 속하는지 확인합니다.
- 검토 중이거나 잠긴 채널이라면 테스트 실행은 보류하는 것이 안전합니다.

## 22. 쇼핑검색 상품그룹 연결

대주제: 연결값

쇼핑검색은 쇼핑몰 비즈채널과 상품그룹의 연결이 맞아야 전송 초안이 안전합니다.

### 구현된 기능
- 쇼핑몰 채널 ID와 상품그룹 ID를 함께 확인합니다.
- 스캔된 상품그룹은 어떤 쇼핑몰 채널에 연결되어 있는지 근거를 함께 가져옵니다.
- 선택한 쇼핑몰 채널과 상품그룹의 연결이 다르면 전송 초안이 차단됩니다.
- 직접 입력한 상품그룹은 스캔 근거가 없으므로 경고 상태로 운영자 검토가 필요합니다.

### 운영자가 확인할 점
- 상품그룹이 고객 쇼핑몰의 올바른 상품 묶음인지 확인합니다.
- 상품그룹 자체는 네이버 콘솔에서 먼저 만들어져 있어야 합니다.

## 23. 오래된 스캔 후보 차단

대주제: 연결값

브랜드나 사이트가 바뀌었는데 예전 계정 후보를 잘못 적용하는 상황을 줄입니다.

### 구현된 기능
- 계정 스캔 결과는 상품 유형, 브랜드, 사이트 기준으로 화면에서 맞는지 판단합니다.
- 입력값이 바뀌면 이전 후보가 현재 작업과 맞지 않을 수 있으므로 숨기거나 다시 스캔하도록 유도합니다.
- 저장 이력에는 어떤 연결값으로 전송 직전 초안을 만들었는지 남습니다.
- 쇼핑검색은 연결 상태를 verified, mismatch, unverified, not applicable로 요약합니다.

### 운영자가 확인할 점
- 다른 광고주의 채널이나 상품그룹이 실수로 적용되지 않는지 확인합니다.

## 24. 초안 검증의 역할

대주제: 전송 직전 초안

승인된 항목과 연결값을 바탕으로 실제 외부 요청 직전 형태를 만들고 다시 검사합니다.

### 구현된 기능
- 승인된 항목만 초안에 들어갑니다. 대기나 보류 항목은 제외됩니다.
- 캠페인, 광고그룹, 키워드, 소재, 쇼핑 광고그룹 같은 요청 단위로 정리됩니다.
- 차단과 경고를 함께 보여 운영자가 무엇을 고쳐야 하는지 알 수 있습니다.
- 저장 이력에서는 초안 목록과 요청 대상, 결과 수를 다시 확인할 수 있습니다.

### 운영자가 확인할 점
- 화면의 전송 직전 초안은 검토용입니다. 생성되었다고 해서 바로 네이버에 보내진 것이 아닙니다.

## 25. 차단과 경고를 읽는 법

대주제: 전송 직전 초안

차단은 진행 불가, 경고는 운영자 판단 필요 상태입니다.

### 구현된 기능
- 차단 예시: 승인 항목 없음, 필수 채널 ID 없음, 쇼핑 채널과 상품그룹 불일치, 중복 실행 위험입니다.
- 경고 예시: 직접 입력한 연결값처럼 검증 근거가 부족한 상태입니다.
- 차단 이력도 저장할 수 있어, 왜 진행하지 못했는지 나중에 확인할 수 있습니다.
- 차단 항목은 승인 수에 포함되어도 실제 테스트 실행 조건을 만족하지 못합니다.

### 운영자가 확인할 점
- 차단 문구가 운영자가 바로 조치할 수 있을 만큼 구체적인지 확인합니다.

## 26. 보호된 테스트 실행 조건

대주제: 보호 실행

테스트 생성/수정은 매우 제한된 조건에서만 가능하도록 잠겨 있습니다.

### 구현된 기능
- 관리자 계정이어야 합니다.
- 이미 저장된 준비 완료 초안이어야 합니다.
- 같은 초안에 실행 결과가 이미 있으면 다시 실행하지 못합니다.
- 확인 문구 TEST_EXECUTION_ONLY가 필요합니다.
- 실패나 외부 오류는 민감값이 제거된 메시지로만 저장됩니다.

### 운영자가 확인할 점
- 라이브 계정 활성화나 삭제가 여기서 가능하면 안 됩니다.
- 검수에서는 실행 자체보다 조건 차단이 제대로 되는지 보는 것이 더 중요합니다.

## 27. 현재 플랫폼이 하지 않는 일

대주제: 보호 실행

기능 설명서에서 가장 중요한 부분은 할 수 없는 일을 분명히 아는 것입니다.

### 구현된 기능
- 라이브 캠페인을 켜지 않습니다.
- 운영 중인 네이버 광고 데이터를 삭제하지 않습니다.
- 성과가 나쁘다는 이유로 자동 입찰이나 자동 예산 변경을 하지 않습니다.
- 쇼핑 상품그룹을 새로 만들거나 쇼핑 피드를 수정하지 않습니다.
- 마스터 리포트 작업 생성/삭제 같은 위험한 성과 수집 방식은 차단합니다.

### 운영자가 확인할 점
- 화면 문구가 자동 집행처럼 오해되지 않는지 확인합니다.

## 28. 계획과 초안 이력 저장

대주제: 저장 이력

검증된 작업은 나중에 다시 볼 수 있도록 승인 상태, 연결값, 초안, 경고를 함께 저장합니다.

### 구현된 기능
- 저장 대상: 입력 요약, 승인 큐, 전송 직전 초안, 연결값, 쇼핑 연결 상태, 운영 근거입니다.
- 부분 저장 실패가 생겨도 가능한 범위에서 계획 ID와 운영 알림을 남겨 복구 가능성을 높입니다.
- 저장 성공 후에는 상세 이력에서 승인 로그와 초안 상태를 다시 확인합니다.
- 내부 식별자와 반복 실행 방지 키는 화면과 다운로드에서 제외됩니다.

### 운영자가 확인할 점
- 저장 후 새로고침해도 핵심 판단 근거가 남아 있는지 확인합니다.
- 선택 기능 일부는 데이터베이스 적용 상태에 따라 readiness에서 먼저 준비 여부를 확인해야 합니다.

## 29. 저장 이력 목록 검색과 필터

대주제: 저장 이력

여러 고객과 세팅 기록이 쌓여도 필요한 이력을 빠르게 찾을 수 있게 합니다.

### 구현된 기능
- 검색: 브랜드, 워크스페이스, 저장자, 사이트 주소로 찾습니다.
- 상품 유형 필터: 전체, 파워링크, 쇼핑검색으로 나눕니다.
- 초안 상태 필터: 준비, 차단, 실패, 실행, 초안 없음으로 확인합니다.
- 쇼핑 연결 필터: verified, mismatch, unverified, not applicable 상태를 구분합니다.
- 필터 링크 복사와 CSV 내보내기로 팀원에게 같은 목록을 공유할 수 있습니다.

### 운영자가 확인할 점
- 실제 운영자가 자주 찾는 조건이 충분한지 확인합니다. 예: 담당자, 고객명, 기간.

## 30. 이력 상세에서 확인하는 것

대주제: 저장 이력

한 번 저장한 계획의 판단 근거와 전송 준비 상태를 상세하게 되짚습니다.

### 구현된 기능
- 요약: 승인 수, 보류 수, 초안 수, 차단 수를 한눈에 봅니다.
- 전송 초안: 요청 대상, 방식, 안전 정보, 결과 수를 확인합니다.
- 실행 연결값: 캠페인, 채널, 쇼핑 상품그룹 연결값을 확인합니다.
- 승인 큐: 각 항목의 승인/보류/대기 상태와 메모를 봅니다.
- 감사 로그: 승인 결정과 운영 이벤트의 안전한 표시값을 확인합니다.

### 운영자가 확인할 점
- Payload JSON 다운로드는 내부 검토용입니다. 공개 리포트에는 원본 본문이 빠져야 정상입니다.

## 31. 운영 리포트와 내보내기

대주제: 리포트와 내보내기

플랫폼 안의 판단 결과를 고객 공유, 내부 인수인계, 품질 검수 자료로 바꿉니다.

### 구현된 기능
- 키워드 CSV: 등록/검토/제외 후보를 표로 검수합니다.
- 승인 CSV: 승인 큐 상태와 메모를 공유합니다.
- 운영 메모: 저장 이력의 핵심 내용을 Markdown 문서로 내려받습니다.
- 관리자 CSV: 사용자, 최근 활동, 계정 스캔, 감사 이벤트, 성과 계획, 공유 링크 현황을 내보냅니다.
- PDF-ready 보고: 화면과 Markdown 자료를 바탕으로 고객 보고용 문서를 만들 수 있습니다.

### 운영자가 확인할 점
- 외부에 보내도 되는 정보와 내부 전용 정보가 섞이지 않았는지 확인합니다.

## 32. 7일 제한 공개 리포트 링크

대주제: 공유 리포트

저장 이력 상세에서 외부 공유용 공개 리포트 링크를 만들고 폐기할 수 있습니다.

### 구현된 기능
- 링크는 7일 제한으로 생성됩니다.
- 생성된 원본 토큰은 저장하지 않고, 확인용으로만 안전한 형태를 보관합니다.
- 필요할 때 링크를 폐기할 수 있습니다.
- 관리자 화면에서는 활성, 만료, 폐기, 접근 상태를 토큰 없이 추적합니다.

### 운영자가 확인할 점
- 공유 링크를 고객에게 보내기 전, 공개 화면에 내부 메모나 원본 요청 본문이 보이지 않는지 확인합니다.
- 공개 링크 자체가 접근 권한이므로 만료와 폐기 절차를 운영 규칙에 포함해야 합니다.

## 33. 공개 리포트에서 제외되는 정보

대주제: 공유 리포트

공개 리포트는 고객 설명에 필요한 요약만 보여주고 내부 운영 원문은 숨깁니다.

### 구현된 기능
- 제외: 원본 전송 본문, 반복 실행 방지 키, 내부 감사 원문, 내부 사용자 식별자, 생성자 내부 정보입니다.
- 포함: 브랜드/사이트 요약, 승인 수, 초안 상태, 쇼핑 연결 요약, 안전하게 정리된 운영 근거입니다.
- 상품그룹 추천과 업종 템플릿 요약은 민감한 내부 원문 없이 표시됩니다.
- 오류 문구는 토큰이나 키처럼 보이는 값을 가린 뒤 표시합니다.

### 운영자가 확인할 점
- 공개 리포트를 비로그인 상태에서 열어보고 고객에게 보여도 되는지 판단합니다.

## 34. 관리자 대시보드 개요

대주제: 관리자

관리자는 사용자와 운영 상태, 외부 조회, 이력 품질을 한 화면에서 점검합니다.

### 구현된 기능
- 사용자 관리: 가입 계정, 권한, 메일 확인, 워크스페이스 연결 상태를 봅니다.
- 운영 헬스: 앱, Supabase, Naver 설정, 선택 기능 준비 상태를 확인합니다.
- 최근 활동: planning run과 execution draft 상태를 추적합니다.
- 운영 알림: 실패, 차단, 설정 누락, 공유 링크 생성/폐기 같은 이벤트를 확인합니다.
- 성과 sync: 읽기 전용 성과 조회 계획과 예약 실행 상태를 봅니다.

### 운영자가 확인할 점
- 관리자 화면은 운영 통제실입니다. 기능이 많으므로 사용자, 상태, 알림, 성과 네 묶음으로 보면 쉽습니다.

## 35. 사용자 관리

대주제: 관리자

초대, 권한, 메일 확인 상태를 관리하고 사용자 목록을 검수 자료로 내보냅니다.

### 구현된 기능
- 이메일 초대: 링크나 토큰을 화면에 노출하지 않고 회원 초대를 기록합니다.
- 권한 변경: member와 admin 역할을 관리합니다.
- 메일 확인 보조: 필요한 경우 확인 상태를 관리자가 보정할 수 있습니다.
- 검색/필터: 권한, 메일 미확인, 로그인 없음, 워크스페이스 없음 상태로 찾습니다.
- CSV: 현재 필터에 맞는 사용자 목록을 내려받습니다.

### 운영자가 확인할 점
- 일반 운영자에게 관리자 권한이 잘못 부여되지 않았는지 확인합니다.

## 36. 운영 헬스와 준비 상태

대주제: 관리자

운영에 필요한 기본 연결과 선택 기능이 준비되어 있는지 안전하게 요약합니다.

### 구현된 기능
- 앱 상태: 기본 서비스가 응답하는지 확인합니다.
- Supabase 준비: 이력, 공유 링크, 쇼핑 연결, 실행 연결값 같은 저장 기능 상태를 봅니다.
- Naver 준비: 외부 조회에 필요한 설정이 갖춰졌는지 값을 노출하지 않고 확인합니다.
- 선택 기능: 일부 테이블이 아직 적용 중이어도 전체 서비스 실패로 보지 않고 준비 상태로 표시합니다.

### 운영자가 확인할 점
- 무엇이 부족한지는 알 수 있지만 실제 키 이름이나 값은 보이지 않아야 합니다.

## 37. 계정 스캔 이력

대주제: 관리자

네이버 계정에서 읽어온 캠페인, 채널, 상품그룹 상태를 운영 감사 자료로 보관합니다.

### 구현된 기능
- 최근 스캔 목록에서 브랜드, 상품 유형, 스캔 시각, 결과 수를 확인합니다.
- 직전 스캔 대비 채널/캠페인/상품그룹 변화 요약을 확인합니다.
- 부분 실패나 경고가 있으면 어떤 범위에서 문제가 있었는지 표시합니다.
- CSV로 내려받아 계정 점검표나 QA 자료에 붙일 수 있습니다.

### 운영자가 확인할 점
- 원본 네이버 인벤토리 전체 배열은 화면과 CSV에 그대로 노출하지 않는 것이 정상입니다.

## 38. 감사 이벤트와 운영 알림

대주제: 관리자

누가 무엇을 했는지, 어떤 운영 실패가 있었는지 확인하는 기록 화면입니다.

### 구현된 기능
- 관리 이벤트: 초대, 권한 변경, 메일 확인 보정 같은 사용자 관리 기록입니다.
- 운영 알림: 저장 실패, 성과 sync 실패/차단, 예약 실행 확인, 공유 링크 생성/폐기 같은 이벤트입니다.
- 필터: 전체, 운영 알림, 초대, 메일 확인, 권한 변경, 기타로 나눠 봅니다.
- CSV: 현재 필터에 맞는 감사 이벤트를 검수 자료로 내려받습니다.
- 원본 변경 전/후 값은 화면 응답에서 제외하고 안전한 요약만 보여줍니다.

### 운영자가 확인할 점
- 운영 실패가 조용히 묻히지 않고 관리자 화면에서 찾을 수 있는지 확인합니다.

## 39. 성과 sync의 목적

대주제: 성과 조회

성과 sync는 광고를 바꾸는 기능이 아니라, 읽기 전용 성과 확인과 운영 추천을 만드는 기능입니다.

### 구현된 기능
- 네이버 성과는 GET 방식의 읽기 전용 조회만 허용합니다.
- 보고서 작업 생성이나 삭제는 계속 차단합니다.
- 원본 성과 데이터 전체는 저장하지 않고, 요약 수치와 추천 요약만 남깁니다.
- 성과 기반 추천도 자동 적용이 아니라 승인 초안 후보로만 표시됩니다.

### 운영자가 확인할 점
- 성과가 나쁘니 자동으로 입찰 변경 같은 흐름이 없어야 합니다.

## 40. 성과 preview와 sync 계획

대주제: 성과 조회

관리자가 캠페인, 광고그룹, 키워드, 소재 ID를 넣어 안전하게 성과 조회를 미리 확인합니다.

### 구현된 기능
- 성과 조회 ID를 입력해야 외부 조회가 시작됩니다. ID가 없으면 요청 전 차단됩니다.
- 파워링크 일별 성과와 쇼핑검색 키워드 성과 범위를 구분해 저장합니다.
- preview 결과는 운영 메모, 추천, 승인 초안 후보로 볼 수 있습니다.
- 저장된 계획은 상태와 범위 필터로 다시 찾고 CSV로 내보낼 수 있습니다.

### 운영자가 확인할 점
- 잘못된 ID, 빈 ID, 권한 없는 사용자의 조회가 요청 전 차단되는지 확인합니다.

## 41. 수동 성과 sync 큐

대주제: 성과 조회

관리자는 저장된 계획 중 안전한 대상만 수동으로 다시 조회할 수 있습니다.

### 구현된 기능
- 대상: 정책상 허용된 planned, failed, ready 상태의 계획입니다.
- 제외: blocked, completed, masterReference 같은 위험하거나 완료된 대상입니다.
- 조회 결과는 completed 또는 failed로 업데이트되고 원본 성과 JSON은 저장하지 않습니다.
- 실패와 차단은 운영 알림으로 남아 관리자 화면에서 확인할 수 있습니다.

### 운영자가 확인할 점
- 수동 sync는 운영자가 다시 조회하는 기능이지, 광고를 수정하는 기능이 아닙니다.

## 42. 예약 실행 정책

대주제: 성과 조회

하루 한 번 정해진 시간에 읽기 전용 성과 조회 계획을 제한적으로 처리합니다.

### 구현된 기능
- 주기: 매일 오전 9시 10분 KST입니다.
- 처리량: 한 번에 최대 3건만 처리합니다.
- 대상: planned 또는 failed 상태입니다.
- 제외: blocked, completed, ready, masterReference는 처리하지 않습니다.
- 자동 재시도 없이 실패 시 운영 알림을 남깁니다.

### 운영자가 확인할 점
- 관리자 화면의 예약 실행 카드에서 켜짐 여부, 비밀값 존재 여부, 대상 건수, 최근 알림을 확인합니다.
- 예약 실행도 읽기 전용 조회이며 광고 생성, 수정, 삭제를 하지 않습니다.
- 하루 최대 3건이므로 backlog가 생기면 수동 정리나 정책 조정이 필요할 수 있습니다.

## 43. 민감정보 보호 방식

대주제: 보안과 데이터

운영자 화면, CSV, 리포트, 오류 메시지에서 민감값이 흘러나오지 않도록 여러 번 가립니다.

### 구현된 기능
- API key, token, Authorization, cookie처럼 보이는 값은 표시 전에 가립니다.
- 환경값 이름과 실제 값은 공개 상태 화면에 노출하지 않습니다.
- 내부 사용자 식별자, 반복 실행 방지 키, 공유 링크 토큰 원문은 화면과 CSV에서 제외합니다.
- 외부 네이버 오류도 그대로 보여주지 않고 안전하게 정리한 문구만 사용합니다.

### 운영자가 확인할 점
- 오류가 났을 때 화면에 긴 토큰, 키, 내부 ID처럼 보이는 문자열이 나오지 않는지 확인합니다.

## 44. 캐시 방지와 잘못된 요청 차단

대주제: 보안과 데이터

보호된 데이터가 브라우저나 서버 캐시에 남지 않도록 응답과 요청 방식을 통제합니다.

### 구현된 기능
- 보호된 조회와 공개 리포트는 최신 상태를 보도록 no-store 정책을 사용합니다.
- 허용되지 않은 방식의 요청은 405 응답으로 명확히 거절합니다.
- 인증이 필요한 화면은 인증이 없으면 401로 막혀야 합니다.
- 클라이언트의 주요 GET 요청도 캐시를 쓰지 않도록 설정되어 있습니다.

### 운영자가 확인할 점
- 로그아웃 후 뒤로가기로 관리자 데이터가 보이지 않는지 확인합니다.
- 지원하지 않는 요청은 데이터를 보여주지 않고 거절되어야 합니다.

## 45. 배포 전 자동 점검

대주제: 검증 체계

코드 변경 후에는 정책 위반이 생기지 않았는지 자동 점검을 통과해야 합니다.

### 구현된 기능
- API 정책 점검: no-store, 405 처리, 민감 응답 제외를 확인합니다.
- 오류 가림 점검: 토큰, 키, 쿠키, 인증 헤더가 화면용 문구로 새지 않는지 확인합니다.
- 성과 안전 점검: 예약 실행 시간, 최대 3건, 읽기 전용 조회, 원본 성과 미저장을 확인합니다.
- 전체 검증: lint와 production build를 통해 화면과 서버 코드가 함께 통과해야 합니다.

### 운영자가 확인할 점
- 운영자가 직접 실행할 필요는 없지만, 배포 전 검증 통과 여부는 릴리스 체크리스트에 포함하는 것이 좋습니다.

## 46. 운영자가 꼭 눌러봐야 할 검수 시나리오

대주제: 사용자 검수

남은 1%는 실제 사용자가 헷갈리는 지점과 업무에 부족한 부분을 찾는 단계입니다.

### 구현된 기능
- 새 브랜드로 파워링크 계획을 만들고 승인, 계정 스캔, 초안 검증, 이력 저장까지 진행합니다.
- 쇼핑검색으로 바꿔 쇼핑몰 채널과 상품그룹 연결 경고/차단을 확인합니다.
- 저장 이력에서 검색, 필터, 상세, 운영 메모, 공유 링크를 확인합니다.
- 관리자에서 사용자, 운영 헬스, 계정 스캔 이력, 운영 알림, 성과 sync 계획을 확인합니다.
- 권한 없는 계정으로 관리자 기능과 외부 조회가 막히는지 확인합니다.

### 운영자가 확인할 점
- 검수 기록은 버그, 문구 개선, 추가 기능, 업무 정책 결정 필요로 나눠 남기면 다음 큐가 빨라집니다.

## 47. 추가 기능 후보를 판단하는 기준

대주제: 사용자 검수

모든 요구를 바로 넣기보다 실제 운영 효율과 안전을 기준으로 우선순위를 정합니다.

### 구현된 기능
- 반복 업무 감소: 같은 클릭이나 복사가 자주 반복되는가?
- 검수 품질 상승: 실수를 줄이거나 판단 근거를 더 잘 보여주는가?
- 고객 설명력 향상: 보고서나 공유 링크가 고객 이해를 높이는가?
- 운영 안전 유지: 새 기능이 라이브 집행, 삭제, 민감값 노출 위험을 만들지 않는가?
- 권한 분리 필요: 관리자와 일반 운영자의 역할을 더 나눠야 하는가?

### 운영자가 확인할 점
- 추천 개선 후보: 온보딩 체크리스트, 상품그룹 매핑 도우미, 보고서 템플릿, 운영 알림 설정, 대량 승인 UX입니다.

## 48. 운영 중 자주 볼 수 있는 상태

대주제: 문제 해결

에러를 기술 문제로만 보지 말고, 어떤 조치가 필요한 상태인지 읽는 것이 중요합니다.

### 구현된 기능
- 승인 필요: 승인된 항목이 없어 전송 직전 초안을 만들 수 없습니다.
- 채널 필요: 비즈채널 또는 쇼핑몰 채널 ID가 아직 연결되지 않았습니다.
- 상품그룹 불일치: 선택한 쇼핑 채널과 상품그룹의 연결 근거가 맞지 않습니다.
- 권한 필요: 관리자 기능을 일반 계정으로 열려고 했습니다.
- 성과 조회 ID 필요: 성과 preview에 사용할 캠페인, 광고그룹, 키워드, 소재 ID가 없습니다.

### 운영자가 확인할 점
- 각 상태가 무엇을 하면 풀리는지까지 안내하는지 확인합니다.

## 49. 플랫폼에서 자주 나오는 쉬운 용어 정리

대주제: 용어 정리

운영자가 기술 용어에 막히지 않도록 화면 용어를 쉬운 말로 정리합니다.

### 구현된 기능
- 작업공간: 한 브랜드 또는 한 광고 운영 묶음을 준비하는 공간입니다.
- 승인 큐: 자동 생성된 후보를 사람이 통과, 보류, 차단 확인하는 목록입니다.
- 전송 직전 초안: 외부로 보내기 바로 전 형태로 정리된 검토용 요청 묶음입니다.
- 계정 스캔: 네이버 계정의 현재 채널, 캠페인, 상품그룹을 읽어오는 일입니다.
- 성과 sync: 광고 성과를 읽어서 요약과 추천을 만드는 일입니다. 광고를 바꾸지는 않습니다.

### 운영자가 확인할 점
- 고객에게 설명할 때는 payload, cron보다 전송 초안, 예약 조회처럼 말하는 편이 좋습니다.

## 50. 검수 후 남길 결과물

대주제: 마무리

사용자 검수가 끝나면 플랫폼 이해뿐 아니라 다음 개발 우선순위까지 정리되어야 합니다.

### 구현된 기능
- 필수 수정: 안전, 권한, 민감정보, 잘못된 저장/공유처럼 바로 고쳐야 하는 항목입니다.
- 업무 개선: 운영자가 더 빠르게 일할 수 있게 하는 필터, 버튼, 설명, 내보내기 개선입니다.
- 정책 결정: 라이브 모드, 역할 분리, 고객 공유 범위처럼 제품 책임자가 정해야 할 항목입니다.
- 추가 자동화: 성과 추천, 보고서, 알림처럼 안전선을 유지하면서 넓힐 수 있는 큐입니다.

### 운영자가 확인할 점
- 현재 완성도의 핵심은 자동 집행이 아니라 승인 전까지 안전하게 멈추는 자동화입니다.
- 사용자 검수 1%는 이 안전한 흐름이 실제 업무에서도 충분히 편한지 확인하는 단계입니다.
